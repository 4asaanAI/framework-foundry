Context
Layaa AI currently uses Relevance AI to host 14 AI agents. The system is limited: no persistent memory, no inter‑agent communication, no task scheduling, no cost control, no self‑improvement. Layaa OS replaces Relevance AI with a fully custom internal platform — a Claude‑equivalent app where all 22 agents are live, interlinked, self‑learning, and capable of executing real tasks autonomously within guardrails. Built for 2 founders now, extensible to future employees.

The platform also includes a pre‑loaded skills library (60 skills accessible via / command) and 8 meta‑skill plugins that add Layaa AI‑specific business context, plus 14 company documents stored in the database and available to every agent.

Browser → VPS: HTTPS only (nginx terminates SSL)

VPS → LLM providers: n8n makes API calls server‑side. Keys stored encrypted in PocketBase settings table (AES‑256, master key in n8n environment). n8n reads keys on each request.

Credential vault: Non‑secret metadata only in credential_vault. Actual keys never stored in PocketBase.

LLM Abstraction Layer ★ (Opus Handoff #1)
Architecture: n8n as LLM Proxy with PocketBase key store

API keys never reach the browser. All LLM calls go through n8n webhooks. Keys are stored encrypted in PocketBase settings table (key = llm_key_<provider>). n8n reads them on each request using its built‑in HTTP Request node with credentials or by fetching from PB.

n8n webhook authentication: Every webhook validates X‑API‑Key header. The key is generated during first‑run (32 random chars) and stored in PB settings (key = n8n_webhook_key). n8n workflow first node is an IF node that checks {{ $json.headers['x-api-key'] }} against the stored key. Mismatch returns 401.

Client‑side layer (src/lib/llm.ts):

sendMessage(request: LLMRequest): Promise<LLMResponse> — POSTs to N8N_WEBHOOK_URL/llm-call with X‑API‑Key header.

streamMessage(request: LLMRequest): AsyncGenerator<LLMStreamChunk> — POSTs to /llm-stream, reads SSE. Includes streaming fallback (on error, retry non‑streaming).

calculateCost(...) — client‑side for UI.

redactCredentials(text) — applies regex patterns before sending.

Streaming fallback code snippet (in streamMessage):

typescript
export async function* streamMessage(req: LLMRequest): AsyncGenerator<LLMStreamChunk> {
  try {
    const response = await fetch(`${N8N_WEBHOOK_URL}/llm-stream`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json', 'X-API-Key': webhookKey },
      body: JSON.stringify(redactContent(req)),
    });
    // ... process SSE stream normally
  } catch (streamError) {
    console.warn('[LLM] Stream failed, falling back to non-streaming:', streamError);
    const response = await sendMessage({ ...req, stream: false });
    yield { type: 'chunk', content: response.content };
    yield { type: 'done', tokens_in: response.tokens_in, tokens_out: response.tokens_out };
  }
}
n8n workflow pseudo‑structure (for Opus):

Workflow 1: /webhook/llm-call (non‑streaming)

Webhook → validate X‑API‑Key → rate limit check (PB rate_limit_{profile} atomic) → read + decrypt LLM key → switch on provider → normalise response → log usage → return.

Workflow 2: /webhook/llm-stream (streaming)

Same as above but with SSE proxy. On stream error, fall back to non‑streaming call.

Workflow 3: /webhook/send-email (Resend)

Validate key → read Resend key → POST to Resend → on error, write to failed_writes.

Workflow 4: /webhook/conversation-end (Memory extraction)

Validate key → call Haiku to extract facts → write to agent_memory → bill to system budget.

Cron Workflow 5: Hourly Health Check

Check DB size → log → trigger compression if >70% → retry failed writes → send budget warnings.

Cron Workflow 6: Daily Backup (3am IST)

POST /api/backups → download ZIP → upload to Backblaze B2 → delete local old backups → verify.

Opus must produce: n8n workflow JSONs (importable) and full src/lib/llm.ts with streaming fallback, tool call normalisation, and credential redaction.

Sage — Memory & Context Keeper ★ (Opus Handoff #3)
File: src/agents/sage.ts

Sage is a background service (no chat UI). Tools:

getRelevantMemory(agentId, topic, limit) — relevance scoring with TF‑IDF‑like weighting

Relevance scoring algorithm (improved from keyword overlap):

Split query into words.

For each memory: score = (matchedWords / totalWords) * IDF_weight + recencyScore*0.3 + categoryBonus*0.2 + confidence*0.1

Compressed memories get 0.5x multiplier.

Return top limit memories.

Dynamic confidence:

On thumbs up for a conversation: increase confidence of memories used in that conversation by +0.05 (clamp max 1.0).

On thumbs down: decrease by -0.1 (clamp min 0.1).

Meaningful content check before extraction:

Budget System (Fixed)
File: src/lib/budget.ts

Atomic increment (fixes race condition):

typescript
await pb.collection("agents").update(agentId, { "budget_used+": tokensIn + tokensOut });
Budget reset (period‑based):

budget_period_start compared with current date.

Kaiser cron updates budget_period_start to now() on reset; budget_used stays as‑is.

Client‑side: effective used = budget_used recorded after budget_period_start.

Loan‑based transfers (not permanent donation):

Add budget_loaned field to agents table.

Transfer: increase recipient's budget_loaned, donor's budget_loaned decreases (negative value allowed).

Effective budget = budget_tokens + budget_loaned.

Loans reset at month end (Kaiser cron sets budget_loaned = 0 for all).

System budget pool:

system_budget_total (default 500,000 tokens), system_budget_used stored in settings.

Memory extraction and Kaiser cron jobs draw from this pool.

Dashboard shows system budget alongside agent budgets.

export async function retryFailedWrites(): Promise<{ retried: number; abandoned: number }> {
  const failed = await pb.collection('failed_writes').getFullList({
    filter: 'retry_count < 5',
    sort: 'created',
  });
  let retried = 0, abandoned = 0;
  for (const record of failed) {
    try {
      await pb.collection(record.collection).create(record.data);
      await pb.collection('failed_writes').delete(record.id);
      retried++;
    } catch (error) {
      const newCount = record.retry_count + 1;
      if (newCount >= 5) {
        await pb.collection('failed_writes').update(record.id, { retry_count: newCount });
        await createNotification('system', 'Failed write abandoned',
          `Write to ${record.collection} failed 5 times: ${record.error}`, 'system');
        abandoned++;
      } else {
        await pb.collection('failed_writes').update(record.id, { retry_count: newCount });
      }
    }
  }
  return { retried, abandoned };
}

export async function getArchivedMessageContent(messageId: string): Promise<string | null> {
  try {
    const archive = await pb.collection('message_archives').getFirstListItem(`message_id = "${messageId}"`);
    return archive.content;
  } catch { return null; }
}
chunkProjectKB(kbId) – File: src/lib/memory.ts

export async function chunkProjectKB(kbId: string): Promise<number> {
  const kb = await pb.collection('project_kbs').getOne(kbId);
  const text = kb.content;
  const CHUNK_SIZE = 8000; // ~2000 tokens
  const OVERLAP = 200;
  const chunks: string[] = [];
  for (let i = 0; i < text.length; i += CHUNK_SIZE - OVERLAP) {
    chunks.push(text.slice(i, i + CHUNK_SIZE));
  }
  for (let idx = 0; idx < chunks.length; idx++) {
    const keywords = extractKeywords(chunks[idx], 10);
    await pb.collection('project_kb_chunks').create({
      project_kb_id: kbId,
      chunk_index: idx,
      content: chunks[idx],
      keywords: keywords,
    });
  }
  return chunks.length;
}

  action: AuditAction,
  actor: string,

  }
}

Rate Limiting Implementation (Concrete)
n8n workflow rate‑check node (first node after webhook):

Query PB settings where key = "rate_limit_{profile_id}".

Parse JSON: { count: number, window_start: string }.

If window_start is >60 seconds ago: reset to { count: 1, window_start: now }.

If count >= 10 and window_start within 60 seconds: return HTTP 429 with Retry-After header.

Else: increment count, update PB setting, proceed.

Client‑side (llm.ts):

On 429, read Retry-After, show toast “Rate limited — wait Ns”, auto‑retry after delay.
